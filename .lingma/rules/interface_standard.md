## 接口复用与分层架构规范

在编写新功能或修改现有代码时，请严格遵循以下原则，以最大化代码复用性、保持职责清晰并避免冗余：

### 1. 优先检索与复用现有接口
- **全面搜索**：在实现新功能前，必须搜索项目中是否已存在功能相同或高度相似的接口/函数。
- **微小区别处理**：若新需求与现有接口仅有参数、返回值或轻微逻辑差异，严禁重新编写完整逻辑。应采用以下策略：
  - **适配器/包装器**：创建轻量级函数，转换参数或返回值以适配新需求。
  - **扩展参数**：在现有函数中增加可选参数来覆盖新场景。
  - **继承/组合**：利用面向对象特性复用核心逻辑。

### 2. 分层设计与职责分离
遵循“高内聚、低耦合”原则，将代码按职责划分为不同层级：

#### A. 通用工具层 (Utils)
- **定义**：存放与业务逻辑无关、跨模块通用的纯技术实现（如日期格式化、加密解密、字符串处理、文件IO等）。
- **组织方式**：
  - **同类场景聚合**：功能相近的工具方法应归入同一个工具类（例如 `DateUtils`, `StringUtils`）。
  - **异类场景隔离**：不同领域的工具应拆分到不同的工具类中，避免产生巨大的“上帝类”。
  - **目录结构建议**：
    utils/
    ├── date_utils.py      # 日期时间相关
    ├── crypto_utils.py    # 加密解密相关
    └── file_utils.py      # 文件操作相关

#### B. 业务处理层 (Handlers/Services)
- **定义**：存放特定业务场景下的逻辑处理、数据组装或流程控制。
- **组织方式**：
  - 按业务领域或功能模块划分处理器（例如 `StockDataHandler`, `UserAuthHandler`）。
  - 处理器应依赖底层 Utils 和基础 API，不应包含重复的基础技术实现。

#### C. 接口/控制器层 (API/Controllers)
- **定义**：负责接收请求、参数校验、调用 Handler/Service 并返回响应。
- **要求**：保持轻薄，仅做流程调度，不包含复杂业务逻辑。

### 3. 重构与提取策略
- **公共逻辑提取**：当发现多个模块存在重复逻辑时，必须评估其归属：
  - 若为通用技术逻辑 -> 提取至 `utils` 目录下的相应工具类。
  - 若为特定业务流程 -> 提取至 `handlers` 或 `services` 目录下的对应处理器。
- **新建目录决策**：
  - 若抽离的类数量较少且属于现有分类，直接放入现有文件。
  - 若形成新的独立功能域或文件过多，可考虑创建新的子目录（如 `utils/financial/`），但需保持目录结构简洁。

### 4. 禁止事项
- ❌ 禁止复制粘贴大块代码并进行少量修改。
- ❌ 禁止在业务逻辑层重复实现已在 Utils 中存在的基础功能。
- ❌ 禁止在不必要的情况下创建新的 API 端点，应优先扩展现有端点。
- ❌ 禁止将不同职责的代码混杂在同一个文件中（如将数据库连接工具与股票分析算法写在一起）。

### 5. 审查清单
- [ ] 是否已搜索过类似功能的实现？
- [ ] 新增代码是否符合分层架构（Utils vs Handler vs API）？
- [ ] 通用工具是否已归位到合适的 Util 类中？
- [ ] 是否存在可以提取为公共方法的重复逻辑？
